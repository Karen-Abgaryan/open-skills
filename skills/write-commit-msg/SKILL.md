# write-commit-msg

## What this skill is and is not

- **Is**: a generator of commit message text for already-staged changes, in a Conventional-Commits-inspired house style that reads as a journal/history entry of what was changed.
- **Is not**: a committer. This skill never runs `git commit`, `git add`, or anything that mutates the repository. Present the drafted message to the user; only commit if they explicitly ask, and as a separate step outside this skill.
- **Is not**: a strict Conventional Commits implementation. Footer trailers, `BREAKING CHANGE:` markers, and the `!` breaking-change indicator are intentionally out of scope by default.
- **Is not**: applicable when nothing is in scope. If default scope (`git diff --cached`) is empty *and* the caller has not provided an explicit override (see "Scope"), stop and ask the user to stage changes or supply a file/hunk list — do not silently fall back to unstaged hunks, untracked files, `git status`, or recent history.
- **Is not**: applicable to merge commits or revert commits.

## Scope

**Default:** review and describe **only the changes currently staged in Git** (`git diff --cached`).

**Caller override:** when the caller hands the skill an explicit list of changed files, paths, or hunks, treat that list as the authoritative scope and describe exactly those — regardless of whether those files are staged, unstaged, or untracked. Do not run `git diff --cached` in this mode, do not extend the scope beyond what the caller named, and do not mix in additional changes "for context."

## Output format

```
<type>[(scope)]: <summary>

<description>
```

## Type

Lowercase. Prefer one of the three primary types, in order:

1. `feat` — introduces a new feature.
2. `fix` — patches a bug.
3. `impr` — improvement of an existing feature (refactor, performance, polish that does not change visible behavior).

Fall back to `docs`, `chore`, `test`, `build`, `ci`, `style`, `revert`, `perf`, or `refactor` only when none of the three primary types fit.

## Scope (optional)

- One word, or a path-like dotted/slashed identifier referring to a package, component, or area of the codebase.
- Allowed characters: lowercase letters, digits, `-`, `_`, `.`, `/`. No spaces, no parentheses, no uppercase.
- Must be predictable for automation (changelog grouping, build/check triggers, file-pattern routing) — prefer stable names matching directory or package boundaries.
- Omit when the change spans the whole codebase or no obvious scope applies.

## Authoring order

Write the **description first**, then derive the **summary** from it.

**Why:** the description is the source-of-record (a journal entry); the summary is its short headline. Drafting the headline first tends to produce a vague, lossy summary that the description then has to repeat or reconcile against.

## Description

- Past tense, passive voice.
- 1–3 statements, ordered from most important.
- A single statement is written as one sentence; multiple statements are formatted as a bullet list.
- At most **30 words** total.
- Include only when the summary alone is insufficient to convey context, motivation, or non-obvious consequence.

## Summary

- A short summarization of the staged changes.
- Past tense.
- At most **8 words**.

## Breaking changes

**By default, skip breaking-change analysis entirely.** Do not inspect the diff for breaking-change signals, do not raise the topic in your reply, and never add `!` after the type/scope or a `BREAKING CHANGE:` footer. Treat breaking-change marking as out of scope unless the user explicitly requests it.

When — and only when — the user explicitly requests it, perform the analysis and follow the rules in [references/conventional-commits.md](references/conventional-commits.md) (load that reference at that point — see "Reference" below).

## Forbidden content

- No mention of AI, bots, coding agents, LLMs, or model names.
- No `Co-authored-by:` lines crediting AI.
- No "generated by" / "with help from" / "assisted by" attributions to any tool or model.

## Result vs. commentary

The drafted commit message is **the only requested result**. Anything else you produce — clarifying questions, warnings, suspicions of a breaking change, scope-mismatch concerns, validation-skipped notes, "FYI" remarks, or any other meta-commentary — belongs **only in your chat reply, clearly separated from the message block**, never inside the message itself.

The commit message text must contain nothing but the commit message: no leading or trailing notes, no parenthetical asides, no "auto-validated: false" markers, no anchor comments, no emoji-tagged annotations. The caller may pipe the message straight into `git commit -F -` or an equivalent mechanism; any extra content lands in the repository history forever.

If something is important enough that the user must see it, put it in the chat reply outside the message block. If it is not important enough for that, drop it.

## Gotchas

- In default mode, inspect **only** `git diff --cached`; do not pull in unstaged hunks, untracked files, or `git status` output as bonus context. When the caller has overridden the scope with an explicit file/hunk list (see "Scope"), inspect exactly that list and nothing else — do not extend beyond what the caller named.
- Past tense + passive voice is **deliberate**, not a mistake to "correct" toward imperative. The commit log is read as a journal of what was changed, not a backlog of instructions to apply. Do not paraphrase toward imperative even if a linter or external convention seems to expect it.
- The `impr` type is **deliberate** because Conventional Commits has no standard type for an improvement to an existing feature that is neither a new capability, a bug fix, nor pure refactor/perf work. Use `impr` for visible-but-not-new improvements; fall back to `refactor` only for code-shape changes with no visible behavior delta, and `perf` only for measurable performance work. If the user is wiring up automation that expects strict Conventional Commits (commitlint, semantic-release, changesets), surface that `impr` will need a custom type allowlist — but only when the user brings up tooling.
- Never run `git commit`, `git add`, `git stage`, `git commit --amend`, or any other repository-mutating command from this skill, even if the user's next message looks like consent to commit. Committing is a separate action outside this skill's scope.
- Do not run `git log`, `git show`, or any other history-inspection command. Everything needed to draft the commit message is defined by this skill's scope rules (staged diff or caller-supplied file/hunk list); prior commits are not input.

## Workflow

1. Determine the scope: if the caller has provided an explicit file/hunk list, use that as-is; otherwise run `git diff --cached` and use the staged changes.
2. Pick the most appropriate type; choose a scope only if it adds automation value.
3. Draft the description (1–3 statements, ≤ 30 words).
4. Summarize it into the first-line summary (≤ 8 words).
5. Validate the message by piping it into the validator on stdin:

   ```bash
   uv run scripts/validate.py <<'MSG'
   <type>(<scope>): <summary>

   <description>
   MSG
   ```

   The validator prints a single-line JSON object to stdout (`{"ok": true}` or `{"ok": false, "errors": [...]}`); each error carries a `code`, a `message`, and a `fix` hint. Diagnostics for humans are mirrored to stderr.

   Word limits are configurable: `--summary-max N` and `--body-max N` (defaults 8 and 30). Exit codes: `0` valid, `1` invalid, `2` usage error.

   **Prerequisites:** `uv` (https://docs.astral.sh/uv/) on `PATH`. The script declares `requires-python = ">=3.10"` via PEP 723 inline metadata; `uv` provisions a matching interpreter automatically.

   **If executing the validator is denied by the user / permission system, skip validation entirely** — do not attempt to re-implement the checks by hand.

6. **When validation runs, the message is acceptable only if it exits 0.** Fix and re-run otherwise.
7. Present the message to the user. Do not run `git commit` — committing is outside this skill's scope.
